这份技术方案旨在为 **100+ 接口、长链路金融业务**提供一套高内聚、低耦合的自动化测试标准。方案核心围绕“**配置驱动、动态生成、状态感知、分层断言**”展开。

------

## 接口自动化测试项目详细技术方案

### 1. 项目整体架构设计

采用分层架构设计，确保 100+ 接口在逻辑变动时，仅需维护最小化单元。

- **测试用例层 (Test Cases)**：基于 `Pytest`。仅负责业务场景描述（Scenario）和预期结果定义。
- **业务流层 (Business Flows)**：原子接口的逻辑编排（如：开户-放款-还款），管理长链路中的数据接力。
- **原子接口层 (Atomic Actions)**：单个接口的封装，负责加载对应的 YAML 模板并调用执行引擎。
- **核心引擎层 (Core Engine)**：
  - **DataEngine**：处理 YAML 懒加载、二级缓存及动态标签解析。
  - **GlobalContext**：线程安全的单例池，管理全局业务变量。
  - **RequestWrapper**：统一处理 HTTP/HTTPS 请求、加解密、鉴权及协议级异常。

------

### 2. 数据驱动与动态参数方案

针对 40-90 个字段的复杂入参，采用“**模板+指令+覆盖**”的策略。

#### 2.1 YAML 模板定义

每个接口对应一个 YAML 文件，文件结构如下：

- **api_info**：路径、方法、基础 Header。
- **payload**：
  - **核心字段**：引用工厂标签 `${gen_id_by_type}`。
  - **非核心字段**：引用通用随机标签 `${rand_str(chn, 20)}`、`${get_mobile}`。
  - **字典字段**：双模设计，支持 `${choice(['01', '02'])}` 或引用公共字典。
- **logic_rules**：定义字段依赖。例如 `id_type=01` 时自动移除 `passport_no`。

#### 2.2 二级缓存渲染机制

1. **一级缓存 (Template Cache)**：`TemplateManager` 首次加载 YAML 时，将其原始字典对象存入 LRU 内存缓存。
2. **二级渲染 (Rendering)**：每次调用前，执行 `deepcopy` 获取模板副本，优先应用用例层的 `overrides` 指定值，再解析剩余动态标签。

------

### 3. 全局上下文与业务闭环

解决 100+ 接口间的数据耦合问题。

- **数据传递**：接口 A 返回的业务主键存入 `GlobalContext.set("apply_no", value)`，接口 B 的 YAML 中通过 `$CONTEXT{apply_no}` 引用。
- **异步任务轮询**：针对金融后端 MQ 处理延迟，设计 `Poller` 机制。在断言前，自动循环查询数据库或状态接口，直到达到目标状态或超时。
- **环境自愈 (Cleanup)**：在测试链路中产生的脏数据（如已占用的证件号），通过 `teardown` 阶段执行逆向 SQL 或注销接口进行清理。

------

### 4. 异常处理与断言体系

针对金融系统特定的自定义状态码与权限拦截逻辑。

#### 4.1 协议与业务双重断言

| **场景类型** | **预期状态码** | **断言逻辑**                                             |
| ------------ | -------------- | -------------------------------------------------------- |
| **正常业务** | HTTP 200       | 校验 `biz_status == "SUCCESS"` 且数据库核心字段落地。    |
| **权限缺失** | HTTP 98880     | 校验 `message` 包含“需要开通A权限”，验证 DB 无新增数据。 |
| **业务互斥** | HTTP 98882     | 校验自定义状态码，匹配“不允许做B业务”文案。              |

#### 4.2 异常分类拦截

- **环境级异常**：如 502、下游超时。触发重试机制，若失败则标记为 `EnvError`。
- **代码级异常**：如 500、JSON 解析失败。标记为 `Bug`。
- **逻辑级异常**：断言不符。记录 Diff 差异，触发熔断（停止当前业务流）。

------

### 5. 数据库与环境健康监控

- **DB 健康预检**：在 `pytest_sessionstart` 阶段对所有涉及的数据库（核心库、字典库）进行心跳测试（`SELECT 1`），不通则立即终止测试任务。
- **自动化连接管理**：`DBClient` 实现自动重连，支持事务回滚模拟。

------

### 6. 报告与可视化 (Allure 定制)

- **链路可视化**：在 Allure 报告中清晰展示 Request -> Response -> DB Check 的完整流水。
- **数据快照**：失败时自动保存当时的 `GlobalContext` 变量池快照，方便复现现场。
- **定制分类**：根据捕获的自定义状态码（9888x），在报告中自动归类为“权限验证点”或“安全拦截点”。

------

### 7. 技术栈选型汇总

- **核心框架**：Python 3.9+ / Pytest
- **请求处理**：Requests / HTTPX (支持异步)
- **数据管理**：PyYAML / Pydantic (类型校验)
- **随机生成**：Faker (定制化生成器)
- **数据库**：PyMySQL / SQLAlchemy
- **报告展示**：Allure

------

这套方案通过**“标签化 YAML”**解决了 40-90 字段的随机化问题，通过**“自定义状态码映射”**解决了金融业务的精准断言，通过**“二级缓存”**解决了 100+ 文件的加载性能。
