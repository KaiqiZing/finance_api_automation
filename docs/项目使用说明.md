# finance_api_automation 项目使用说明

本文档说明如何安装依赖、配置环境、运行测试与扩展维护。适用对象：测试开发、接口自动化维护人员。

---

## 1. 项目简介

本项目为金融业务**接口自动化测试**框架，采用五层架构：**用例层（tests）→ 业务流（business）→ 原子接口（api）→ 核心引擎（core）→ 数据模板（data/templates）**。特点包括：

- **YAML 模板 + 动态标签**：减少硬编码入参，支持 `${get_mobile}`、`$CONTEXT{key}` 等表达式。
- **全局上下文 GlobalContext**：长链路场景下跨接口传递申请单号、账户号等变量。
- **RequestWrapper**：统一 HTTP 与金融类业务状态码（如 98880）的异常映射。
- **Pytest + Allure + Coverage**：标准测试与报告、覆盖率输出目录见下文。

---

## 2. 环境要求

- **Python**：建议 3.11 及以上（需与 `requirements.txt` 中依赖兼容）。
- **操作系统**：macOS / Linux / Windows（路径以 `pathlib` 处理，避免手写 `../`）。
- **可选**：安装 [Allure Commandline](https://docs.qameta.io/allure/) 后可使用 `run.py --report` 本地打开 HTML 报告。

---

## 3. 安装

在项目根目录执行：

```bash
cd finance_api_automation
pip install -r requirements.txt
```

若使用虚拟环境（推荐）：

```bash
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

## 4. 配置说明

### 4.1 环境与 API

环境由环境变量 **`TEST_ENV`** 控制，取值为 `dev` 或 `test`，对应文件：

- `config/env_dev.yaml`
- `config/env_test.yaml`

主要字段：

| 配置段 | 含义 |
|--------|------|
| `api.base_url` | 被测服务根地址 |
| `api.timeout` / `max_retries` / `verify_ssl` | 请求超时、重试、是否校验证书 |
| `api.extra_headers` | 附加请求头 |
| `auth` | 登录路径与默认账号（用于 `LoginAPI`） |
| `databases` | 多数据源别名，供 `DBClient` 与断言轮询使用 |
| `poller` | 全局轮询默认超时/间隔（业务代码可按需覆盖） |

**敏感信息**：密码、数据库密码等建议通过 CI/CD 密钥或本地环境变量注入，勿将真实密钥提交到仓库；可在后续扩展中用 `${ENV_VAR}` 替换 YAML 中的占位值（当前 yaml 为示例占位）。

### 4.2 数据库与健康检查

`tests/conftest.py` 中 **`db_health_check`** 会在**整个会话开始前**对所有 `databases` 别名执行 `SELECT 1`。任一连不通时，pytest 会直接退出（避免大批量用例在无效环境空跑）。

使用前请将 `env_*.yaml` 中的 `host` / `port` / `user` / `password` / `database` 改为**可连通的测试库**，并保证测试账号有对应库权限。

**本地无数据库时**：需要临时改造 `conftest.py`（例如增加开关跳过 DB 注册与健康检查），否则无法正常收集/执行用例。

### 4.3 运行前设置环境

使用启动脚本时会自动设置 `TEST_ENV`：

```bash
python3 run.py --env test
```

直接使用 pytest 时，请自行导出：

```bash
export TEST_ENV=test   # 或 dev
pytest
```

## 5. 运行测试

### 5.1 推荐：统一入口 `run.py`

项目根目录：

```bash
python3 run.py                          # 默认 TEST_ENV=test，跑全部 tests 下用例
python3 run.py --env dev                # 使用开发环境配置
python3 run.py --mark smoke             # 仅运行打了 @pytest.mark.smoke 的用例
python3 run.py --mark account -n 4     # 账户相关用例，4 进程并行（需 pytest-xdist）
python3 run.py --failfast               # 首个失败即停止
python3 run.py --report                 # 结束后尝试生成并打开 Allure（需本机安装 allure 命令）
```

### 5.2 直接使用 pytest

```bash
export TEST_ENV=test
pytest tests/test_account/test_register.py -v
pytest -m "smoke" -n 2
```

### 5.3 pytest.ini 中的行为摘要

- 结果与附件：**Allure** 原始数据目录为 `outputs/allure_results`（每次默认清空再写入，见 `pytest.ini`）。
- **覆盖率**：统计 `api`、`business`、`core`，HTML 报告输出到 `outputs/coverage_html`。
- **日志**：控制台 INFO；`outputs/logs/pytest.log` 为 DEBUG 级别（配置见 `pytest.ini与 loguru` 双轨时以实际安装为准）。

### 5.4 Pytest 标记（markers）

在 `pytest.ini` 中已注册，可按标签筛选：

| 标记 | 用途 |
|------|------|
| `smoke` | 冒烟 |
| `regression` | 回归 |
| `account` / `payment` | 按业务模块 |
| `scenario` | 跨模块长链路 |
| `slow` | 耗时较长 |

## 6. Allure 报告

1. 执行测试后，`outputs/allure_results` 下会生成原始数据。
2. 生成静态站点（需安装 Allure CLI）：

```bash
allure generate outputs/allure_results -o report --clean
allure open report
```

失败用例可通过 hook 附加 **GlobalContext 快照**（JSON），便于对照当时上下文变量。

## 7. 目录结构速查

| 路径 | 职责 |
|------|------|
| `api/` | 原子接口：`BaseAPI` + 各模块 `*_api.py` |
| `business/` | 业务流：编排多个 API，写入/读取 `GlobalContext` |
| `core/` | `TemplateManager`、`DataEngine`、`RequestWrapper`、`Validator`、`GlobalContext` 等 |
| `config/` | 环境 YAML 与 `settings.py` 加载 |
| `data/templates/<模块>/` | 接口 YAML 模板（与代码中 module 名一致） |
| `data/schemas/` | JSON Schema，供契约断言 |
| `data/common_dicts.yaml` | 公共枚举示例池 |
| `tests/` | `conftest.py` 与各业务子目录 |
| `utils/` | 日志、DB、身份工厂等 |
| `outputs/` | Allure 中间结果、覆盖率、日志等生成物 |
| `report/` | 可选：Allure generate 输出目录（与 `run.py --report` 一致时使用） |

## 8. 扩展指南

### 8.1 新增接口

1. 在 `data/templates/<业务模块>/` 新建 `xxx.yaml`（含 `api_info`、`payload`，可选 `logic_rules`）。
2. 在 `api/<模块>/` 新建类继承 `BaseAPI`，调用 `_build_payload("模块目录名", "模板名", overrides)` 后通过 `self._wrapper` 发请求。
3. 如需 JSON 契约，在 `data/schemas/` 增加对应 `.json`，断言处调用 `Validator.assert_schema(body, "schema_name")`（不含 `.json` 后缀）。

### 8.2 覆盖模板字段（overrides）

`_build_payload` 的 `overrides` 使用**点分路径**写在整个模板字典上，例如覆盖请求体字段应写：

```python
overrides = {"payload.cust_name": "张三", "payload.id_type": "01"}
```

仅当键与模板**根结构**一致时才可省略前缀；推荐统一使用 `payload.<字段名>`，与同目录下 `BaseAPI` 文档保持一致。

### 8.3 动态标签（DataEngine）

常用标签包括：`${get_mobile}`、`${get_name}`、`${get_id_card}`、`${rand_str(chn,4)}`、`${choice([...])}`、`${uuid}`、`${timestamp}`、`$CONTEXT{apply_no}` 等。新增标签在 `core/data_engine.py` 的 `_dispatch` 中扩展。

### 8.4 新增用例

- 文件/函数命名符合 pytest 规则：`test_*.py`、`test_*`。
- 使用 `@pytest.mark.*` 与 `@allure.feature` / `@allure.story` 便于报告分层。
- 长链路优先调用 `business/` 中已有 flow，避免在 `tests/` 里堆叠过多接口细节。

## 9. 常见问题

**Q：一运行就提示数据库健康检查失败？**  
A：检查 `config/env_*.yaml` 中数据库是否可达、防火墙与账号权限；或按 4.2 节在本地暂时跳过该 fixture（仅限开发自测）。

**Q：实际接口路径与模板里不一致？**  
A：修改对应 API 类中的 `self._wrapper.post/get(...)` 路径，或把路径集中到配置/YAML 的 `api_info` 并在代码中读取（需少量改造）。

**Q：98880 等状态码没有按预期抛出 Python 异常？**  
A：确认后端返回 JSON 中业务码字段名是否为 `code` 或 `biz_code`（见 `core/request_wrapper.py`），若字段名不一致需对齐或扩展解析逻辑。

**Q：并行执行（`-n`）时上下文串用？**  
A：`GlobalContext` 已用锁保护；若多进程下仍有问题，需避免在 session 级 fixture 中写入会被多个 worker 共享的可变全局状态，或改为进程内隔离策略。
